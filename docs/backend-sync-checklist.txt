  KHU Nect (가칭) — 프론트 기준 백엔드 연동·회의 체크리스트
  작성일: 2026-03-29 (프론트 현재 구현 기준, API 미연동·로컬 목 상태)
================================================================================

[1] 전제 / 현재 프론트 상태
--------------------------------------------------------------------------------
  - 실제 HTTP/WebSocket 연동 없음. 인증·프로필·시간표·채팅·DM은 대부분
    localStorage + 클라이언트 상태로 동작.
  - 백엔드와 맞출 때: 엔드포인트 설계, 인증 방식, 실시간(폴링 vs SSE/WebSocket),
    ID/시간 형식, 에러 코드 규약을 우선 확정하는 것이 좋음.


[2] 인증·세션
--------------------------------------------------------------------------------
  - 프론트 User 모델 (목): id, email, nickname, department, grade
  - 로그인 시 전체 UserProfile 객체를 클라이언트에 보관(현재).
  - 회의 안건
    · 로그인 API: 이메일/비번? 학교 SSO? 데모 계정만?
    · 응답: access/refresh 토큰 vs 세션 쿠키 vs JWT 페이로드에 무엇을 넣을지
    · user id: 문자열 UUID 권장 (프론트 전역 user.id로 사용 중)
    · 로그아웃·토큰 만료·401 시 프론트 리다이렉트(/ 등) 동작 합의


[3] 프로필 (온보딩·마이페이지)
--------------------------------------------------------------------------------
  - Profile = UserProfile + (mbti, bio, interests[], hobbies[])
  - interests: { id, label, category: "hobby"|"study"|"lifestyle" }[]
  - hobbies: string[] — 공강 매칭 필터와 동일 소스로 쓰는 중
  - 회의 안건
    · GET/PATCH /users/me 또는 /profile 단일 리소스로 통일할지
    · interests/hobbies 서버 검증(개수 제한, 금칙어 등)
    · 학과·학년(grade)은 코드 테이블 vs 자유 텍스트


[4] 시간표
--------------------------------------------------------------------------------
  - 과목 1건 = TimetableCourse
    · id, name, professor, days[] (요일 문자열 예: "월","화"),
      startTime, endTime (HH:mm), location, type: "major"|"general"
  - 사용자별 시간표는 userId 기준으로 구분 (프론트: khu-nect_timetable_{userId})
  - 회의 안건
    · 시간표 CRUD API (전체 교체 vs 항목 단위)
    · 서버에 “강의실 시간표” 마스터가 있으면: 클라이언트 id와 서버 courseOfferingId 매핑
    · 요일/시간대 타임존·학기 단위(학기 변경 시 초기화 정책)
    · 공강 계산: 서버에서 슬롯 계산할지, 프론트만 할지 (매칭 품질·부하)


[5] 수업 채팅 (과목 단위 오픈채팅)
--------------------------------------------------------------------------------
  - courseId(= 시간표 과목 id)별 메시지 목록
  - 메시지 종류: 일반 / 질문 모드 / 같이 앉기(sitTogether)
  - 같이 앉기: sitRequestId, expiresAt(ISO). 프론트는 약 5분 TTL 후 목록에서 숨김
  - 회의 안건
    · REST 목록 + 페이지네이션 vs WebSocket 실시간
    · 같이 앉기: 수락 시 1:1 DM 방 생성 — 서버가 sitRequestId로 매칭 확정 API 필요할 수 있음
    · 삭제/신고/관리자 가림 정책


[6] 1:1 DM
--------------------------------------------------------------------------------
  - 방(DmRoom): id, participantIds[2], messages[], createdAt
  - 선택 메타: sourceCourseId, matchSource: "sit" | "freeSlot" | "classMatch"
  - 메시지: id, senderUserId, content, createdAt(ISO)
  - 읽음: 프론트는 별도 localStorage(khu-nect_dm_read_state) — 서버와 동기화 시
    lastReadMessageId 또는 readAt per user 필요
  - 회의 안건
    · 방 생성 트리거(같이 앉기 수락, 공강 매칭, 수업 매칭)별 API 또는 통합
    · 메시지 전송·목록·읽음 처리
    · 푸시 알림 vs 인앱만 (매칭 성공 시)


[7] 매칭 성공 “알림” (현재: 로컬 전용)
--------------------------------------------------------------------------------
  - 매칭 시 양쪽 userId에 roomId + 상대 표시 이름(peerLabel) 저장 (로컬)
  - 레이아웃에서 모달로 “매칭 성공” 표시
  - 회의 안건
    · 서버 푸시/인앱 알림으로 대체 시: 알림 타입, 읽음 처리, 딥링크(/home/chat?dm=)


[8] 공강 매칭
--------------------------------------------------------------------------------
  - 취미(hobbies)·시간대(목 데이터) 기반 카드 스와이프형 UI
  - 수락 시 createRoomFromFreeSlotMatch — poster/accepter userId, 표시 라벨
  - 회의 안건
    · 후보 목록 생성 규칙(시간표 공강 겹침, 취미, 거리 등)
    · 매칭 큐/한도/중복 매칭 방지
    · 실시간 대기 인원 vs 배치 작업


[9] 수업 매칭
--------------------------------------------------------------------------------
  - 과목 단위 키: subjectKey = name + "\0" + professor (과목명+교수 동일 = 동일 수업)
  - 내 시간표와 겹치는 수가 많은 사용자 순 정렬 (현재 데모 5명 목만)
  - 대화하기 시 classMatch 소스로 DM 방 생성
  - 회의 안건
    · 동일 수업 판정을 서버 courseOfferingId로 할지, name+professor로 할지
    · 추천 리스트 API(페이지네이스·정렬·차단 사용자 제외)
    · 개인정보 노출 수준(학과만 vs 닉네임까지)


[10] 같이 앉기 → DM 연동
--------------------------------------------------------------------------------
  - 수업 채팅에서 같이 앉기 수락 시 DM 방 + matchSource "sit" + sourceCourseId
  - 회의 안건
    · 수락 API 요청 본문: courseId, sitRequestId, 양쪽 userId
    · 이미 방이 있으면 재사용 vs 신규 방 정책


[11] 데모·테스트
--------------------------------------------------------------------------------
  - 데모 계정 5명: demo-user-1 ~ demo-user-5 (이메일 demo1@khu.ac.kr ~)
  - 수업 매칭 목록은 데모 전용으로 제한해 둔 상태 — 실서비스 시 일반 사용자도
    동일 API로 받을지 결정 필요
  - 회의 안건
    · 스테이징 시드 데이터(시간표 동일 과목) 공유


[12] 공통 기술
--------------------------------------------------------------------------------
  - 날짜/시간: ISO 8601 문자열 권장 (프론트 그대로 표시·정렬)
  - 에러: HTTP 상태 + { code, message } 같은 공통 바디
  - CORS, 쿠키 도메인, 모바일 웹뷰 대비
  - Rate limiting (채팅·매칭 스팸 방지)


[13] 프론트에서 다음 단계로 할 일 (백엔드 스펙 확정 후)
--------------------------------------------------------------------------------
  - api 클라이언트 레이어 추가 (fetch/axios + 인터셉터)
  - Context를 localStorage → 서버 동기화로 점진 교체
  - 로그인 후 토큰 저장 위치(httpOnly 쿠키 vs memory+refresh)
