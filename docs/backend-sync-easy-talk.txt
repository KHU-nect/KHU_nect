백엔드랑 맞추면 좋은 것들 (팀에서 바로 말하기 쉽게 정리)
작성: 2026-03-29

※ 회의 순서 제안: ① 로그인 → ② 프로필 → ③ 시간표 → ④ 수업 채팅 → ⑤ 1:1 대화 → ⑥ 매칭
  (기획/디자인은 ⑦ 디자인·화면·카피를 따로 잡아도 좋아요)

────────────────────────────────────────
먼저 (공통)
────────────────────────────────────────
• 지금은 서버 없이 브라우저 저장(localStorage)으로만 돌아가요. 아래 항목 정해지면 API 붙이면 됩니다.


================================================================================
① 로그인
================================================================================
[ ] 로그인 방식 정하기 (이메일/비번, 학교 SSO, 테스트 전용 등)
[ ] 로그인 성공 시 내려줄 것 (토큰, 쿠키, 사용자 정보 범위)
[ ] 사용자 id 형식 (문자열·UUID 등 — 프론트 전역에서 user.id로 씀)
[ ] 로그아웃 / 토큰 만료 / 401 일 때 화면 이동(예: 로그인 페이지)


================================================================================
② 프로필
================================================================================
[ ] 저장·조회·수정 API (닉네임, 학과, 학년, 자기소개 등)
[ ] 취미·관심사(hobbies / interests) 구조 합의 — 공강 매칭과 같은 데이터 쓰는 중
[ ] 학과·학년: 코드表 vs 자유 입력, 수정 제한 여부


================================================================================
③ 시간표
================================================================================
[ ] 수업 한 건 필드 합의 (과목명, 교수, 요일들, 시작·끝 시간, 강의실, 전공/교양 등)
[ ] 저장 방식 (전체 교체 vs 과목 단위 CRUD)
[ ] 학교 수업 DB와 연결 시 — 앱 쪽 과목 id ↔ 서버 courseOfferingId 매핑
[ ] 학기·타임존 — 학기 바뀔 때 초기화 정책
[ ] 공강 계산을 서버에서 할지, 클라이언트만 할지 (매칭 품질·부하)


================================================================================
④ 수업 채팅 (같은 수업 듣는 사람들 오픈 채팅)
================================================================================
[ ] 과목(courseId)별 채팅 — 메시지 목록·전송 API
[ ] 메시지 종류: 일반 / 질문 모드 / 같이 앉기 요청
[ ] 실시간: REST 폴링 vs WebSocket/SSE
[ ] 같이 앉기: 만료 시간(프론트는 대략 수 분 후 숨김) — 서버 정책 맞추기
[ ] 삭제·신고·운영 정책(필요 시)


================================================================================
⑤ 1:1 대화 (DM)
================================================================================
[ ] 방 생성·조회 — 참가자 2명, 메시지 목록·전송
[ ] 방 메타: 어디서 열렸는지 (같이 앉기 / 공강 매칭 / 수업 매칭), 연결 수업 id 등
[ ] 같이 앉기 수락 시 1:1 방 열기 — 요청 본문(courseId, 요청 id, 양쪽 userId 등)
[ ] 이미 둘 사이 방이 있으면 새로 만들지 않고 이어줄지
[ ] 읽음 처리 (지금은 로컬만 — lastRead, readAt 등 서버 스펙)


================================================================================
⑥ 매칭
================================================================================
[ ] 공강 매칭 — 후보 추천 규칙(공강 겹침, 취미, 차단 등), 수락 시 1:1 방 연결
[ ] 수업 매칭 — “같은 수업” 정의(과목명+교수 vs 수업 코드), 정렬·페이지네이션, 노출 정보 수준
[ ] 매칭 성공 알림 — 지금은 앱 안 모달만; 푸시/알림함 쓸 때 딥링크(예: /home/chat?dm=)
[ ] 스팸·남용 방지 (같은 상대에게 반복 매칭/메시지 제한 등)
[ ] 데모 계정 전용 테스트 범위 vs 실서비스 API 동일 여부

(참고) 시간 형식은 ISO 8601 문자열로 맞추면 프론트 표시·정렬이 편합니다.
(참고) 모바일 웹·추후 앱 대비 CORS, 쿠키 도메인, API 주소도 한 번 짚으면 좋아요.


================================================================================
⑦ 디자인·화면·카피 (백엔드와 별도로 짚으면 좋은 것)
================================================================================
※ 지금 화면 기준으로 “불일치·빈 부분”을 정리했어요. 기획/디자인 회의 때 같이 보면 좋아요.

[ ] 닉네임 쓰는 곳이 적음 — 온보딩·마이 상단 정도이고, 매칭·DM에서는 “OO학과 쿠옹이”처럼
    학과 위주로 보여주는 경우가 많아요. 닉네임을 더 쓸지, 아니면 학과+쿠옹이로 통일할지
[ ] 소개글(bio) — 데이터에는 있을 수 있지만, 본인이 소개를 적거나 고치는 화면은 없어요.
    매칭 카드·상대 프로필에는 목업만 보이는 구간이 있어요. “내 소개” 입력/수정을 넣을지
[ ] 학과·학년 — 처음 프로필 설정에서만 고르고, 이후에는 수정하는 UI가 없어요.
    전과·복학 등 대비해 마이페이지에서 바꿀 수 있게 할지, 수정 주기·인증 필요 여부
[ ] 취미 — 마이페이지에서만 고치는 흐름이에요. 온보딩에서 같이 받을지, 우선순위는 어디인지
[ ] 아바타(사자) 색 — 학과 이름으로 색을 정하는데, 목록에 없는 학과는 회색으로 나가요.
    학과 수가 많으면 색 규칙을 어떻게 할지 (랜덤, 단과대 색, 기본 팔레트 등)
[ ] 프로필 팝업(다른 사람) — “메시지 보내기” 등 버튼이 실제 플로우와 연결돼 있는지 한 번씩 확인
    (화면만 있고 동작이 다른 버튼이 있을 수 있어요)
[ ] 용어 통일 — “쿠옹이”, “OO학과 쿠옹이”, 닉네임, 채팅/대화/쪽지 등 사용자에게 보이는 말 맞추기
[ ] 빈 상태·에러 — 시간표 없음, 매칭 없음, 채팅 없을 때 문구·일러스트 톤 맞추기
